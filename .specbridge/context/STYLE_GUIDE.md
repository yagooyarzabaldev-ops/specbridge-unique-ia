# Style Guide

## Documentation Style

Documentation must be:

- direct
- operational
- auditable
- explicit
- free of vague promises
- aligned with Spec Driven Development

## Engineering Style

When implementation begins, code should be:

- deterministic where possible
- small in scope
- testable
- policy-aware
- easy to review
- aligned with the active execution contract

## Agent Behavior Style

Agents should:

- report evidence instead of confidence
- avoid speculative abstractions
- avoid unrelated refactors
- keep diffs focused
- stop at policy boundaries
- complete ordinary implementation work without step-by-step permission requests

## Product Language

Preferred terms:

- SpecBridge
- Vibe Autopilot
- execution contract
- context package
- policy engine
- risk rules
- final report

Avoid terms that imply unrestricted control, unsafe shell access, or raw conversation dumping.
